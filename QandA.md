## Q1. 勝敗判定のタイミングと「2日目」の到達可能性 [解決済み → v0.3-draftでワンナイト人狼化（7章, 8章, 9章）]

**v0.3追記**: v0.2では「初日占い」を追加するワークアラウンドで対応したが、根本解決のため、v0.3-draftで初期版を**ワンナイト人狼**（夜の襲撃なし、生存者数による勝敗判定を廃止し、処刑結果のみで勝敗決定）に変更した。これにより1試合は夜フェーズ→昼フェーズ→処刑→勝敗判定の1周で必ず完結する設計となり、「2日目に到達しない」という問題はワークアラウンドではなく仕様そのものとして解消された。

- 該当: 7.2 (L132), 8. ゲーム進行 (L136-152)
- 4人村（人狼1・占い師1・村人2）で、7.2「人狼が生存したまま生存者数が2人以下」を文字通り適用すると、
  - Day1処刑で人狼が処刑されなければ生存者は3人 → 勝敗判定で継続
  - Night1で人狼が1名襲撃すると生存者は2人、かつ人狼生存 → 即人狼陣営勝利
  - 結果として、Day1で人狼を処刑できない限り、必ずNight1終了時点で人狼勝利が確定し、8章にある「2日目以降」のフェーズには実質到達しません。
- 占い師の占い結果（9. の「過去の占い結果」）も、そもそも2日目が発生しないなら次のDayフェーズで活用される場面がなく、仕様上の想定と矛盾している可能性があります。
- また、Night内の処理順（8.人狼の襲撃 → 9.占い師の占い → 10.勝敗判定）で、占いは襲撃結果確定後に行うのか、襲撃と同時（襲撃前の生存者情報を基準）に行うのかも未定義です。
- **確認したいこと**: 7.2の勝利条件は4人村用にこのままでよいか（＝実質1〜2ターンで終わる短期決戦を意図している）、それとも「生存者数が2人以下」は将来の6人・8人村向けの汎用ルールで、4人村では別の閾値（例: 人狼数 ≧ 村人陣営数）にすべきか。

## Q2. 昼フェーズにおける発言・投票の情報同時性 [解決済み → SPEC.md 10章]

- 該当: 9. 昼フェーズ (L161)
- 「生存している各プレイヤーが1回ずつ発言し、同時に投票先を決める」とありますが、11.1のJSON出力形式では `speech` と `vote` が同一レスポンス内で同時に生成されます。
- そのため、各AIは「自分より前に発言した他プレイヤーの当日の発言内容」を見た上で投票できるのか、それとも他プレイヤーの当日発言を一切見ずに自分の発言と投票を決めるのか（＝実質は前日までのログのみを根拠にした同時提出）が不明です。
- 呼び出し順（プレイヤーリスト順？ランダム？）も未定義です。
- **確認したいこと**: 発言は逐次公開（先に発言した人のログを後続プレイヤーに見せる）か、全員完全同時（当日の他人の発言は見えない）か。呼び出し順序の決め方は固定か毎タームシャッフルか。

## Q3. JSON構文が正しくても内容が不正な場合の扱い [解決済み → SPEC.md 12.2章]

- 該当: 12. JSONパース失敗時の処理 (L227-242), 13. 投票ルール
- 12章はJSONとして「構文的に」パースできない場合のフォールバックのみを定義しています。
- 構文的には正しいが内容が不正なケース（例: 存在しないプレイヤー名への投票・襲撃・占い、死亡者への投票、人狼が自分自身を`target`に指定、必須キー欠落、型不一致）の扱いが未定義です。
- **確認したいこと**: これらの意味的不正も12章のフォールバック（ランダム選択等）に含めてよいか。含める場合、フェーズごとの許容値チェック（生存者かどうか、自分以外かどうか等）をどこで実施するか（ゲームエンジン側でバリデーションしてから使う、という理解でよいか）。

## Q4. 秘密情報の技術的な隔離手段 [解決済み → SPEC.md 16.2章]

## Q12. `seer_result` / 占い結果の構造と意味 [解決済み → v0.4-draftに反映]

- 該当: 9章、17.2、17.3（`seer_result: null` の例のみ）
- 問題:
  1. `game_state.json` の `seer_result` が `null` 以外でどんな形になるか未定義。占い対象・判定結果・占い師を保持できないと、昼プロンプト生成（10.3）や `results.md` の占い結果出力を一意に実装できない。
  2. 占い師が知る「結果」の意味も未定義。対象の**正確な役職名**（`werewolf` / `seer` / `villager`）を返すのか、ワンナイト系でよくある**人狼 / 人間（非人狼）の二値**なのか。
- **確認したいこと**:
  - `seer_result` を例えば `{"seer":"Codex","target":"Grok","result":"werewolf"}` のような構造でよいか。
  - `result` は正確な役職か、人狼/人間の二値か。
- **決定した方針**:
  - `seer_result` は `{"seer":"Codex","target":"Grok","result":"werewolf"}` のような構造とする。
  - 占い結果は正確な役職名ではなく、人狼 / 人間の二値（`werewolf` または `human`）のみとする。`human` は非人狼を意味し、占い師と村人の区別はしない。

## Q13. 投票ラウンドの「同時決定」と公開タイミング [解決済み → v0.4-draftに反映]

- 該当: 10.2、17.1
- 問題: 「全プレイヤーが同時に投票先を決定する」とあるが、各票をいつ `public_log.md` に載せるかが未定義。CLI呼び出しは実際には逐次になるため、先に返ってきた票をログへ追記して後続AIに渡す実装にすると、「同時投票」の意図に反する。
- **確認したいこと**: 全プレイヤーの投票応答を非公開で収集・確定したあと、一括で `public_log.md` に公開する理解でよいか。
- **決定した方針**:
  - 投票ラウンドは同時投票扱いにする。実装上は逐次呼び出しでも、全員に同一の「発言ラウンド終了時点の public_log.md」を渡す。
  - 投票結果は全員分を内部で収集してから public_log.md に一括公開し、先に返ってきた票を後続プレイヤーには見せない。

## Q14. JSON検証の厳密さ（追加キー・空文字・前後テキスト） [解決済み → v0.4-draftに反映]

- 該当: 11章、12.1、12.2
- 問題: 必須キー欠落・型不一致・不正プレイヤー名は定義されているが、次が未定義:
  - 定義外キーの有無（`additionalProperties`）
  - 空文字（`""`）の `speech` / `vote` / `target` / `reason`
  - JSON前後の説明文、Markdownコードフェンス（\`\`\`json ... \`\`\`）の許容可否
- CLIごとに応答形式がぶれやすく、実装者ごとに検証結果が変わりうる。
- **確認したいこと**: 応答全体を単一JSONオブジェクトとして厳密解析し、定義外キー・空文字・前後テキストは不正（フォールバック）とするか。それともコードフェンス除去などの緩いパースを許すか。
- **決定した方針**:
  - Phase 1では厳密JSONパースとする。応答全体が単一JSONオブジェクトでなければ不正扱いとする。
  - 定義外キー、必須キー欠落、空文字、null、型不一致、前後テキスト、Markdownコードフェンスはすべて不正（フォールバック対象）扱いとする。（緩いパースはPhase 3以降の拡張候補とする）

## Q15. `logs/.../raw/` のファイル命名規則 [解決済み → v0.4-draftに反映]

- 該当: 12.1、12.2、14章
- 問題: 生応答を `raw/` に保存するとあるが、ファイル名規則がない。同一プレイヤーが夜・発言・投票で複数回失敗した場合の上書き可否が不明。
- **確認したいこと**: 試合内の失敗応答をすべて個別保存する必要があるか。
- **決定した方針**:
  - `raw/` のファイル命名規則は `raw/{seq:02d}_{phase}_{player}_{error_type}.txt` とする。
  - `phase` は `night` / `speech` / `vote`。
  - `error_type` は `syntax` / `semantic` / `timeout` / `cli`。
  - 同一試合内の失敗応答はすべて個別保存し、上書きしない。

## Q16. `--seed` の適用範囲と再現性 [解決済み → v0.4-draftに反映]

- 該当: 12.3、13章、15.4
- 問題: `--seed` の存在は定義されているが、次が未定義:
  - 役職割当
  - 同票時の処刑対象
  - 投票失敗・占い失敗時のランダムフォールバック
  - `--games N` 時の試合間での乱数消費順
- **確認したいこと**: 同一コマンドライン・同一設定で完全に同じ試合結果を再現することをPhase 1の要件とするか。
- **決定した方針**:
  - `--seed` 指定時はPhase 1で完全再現可能にする。
  - 単一の乱数生成器を使い、役職割当、dry-run占い先、dry-run投票先、フォールバック、同票処刑対象をすべて同じ乱数生成器で順次処理する。
  - `--games N --seed X` は毎回同じN試合を再現できるようにする。

## Q18. 試合ディレクトリ番号の採番ルール [解決済み → v0.4-draftに反映]

- 該当: 14章「既存の試合ディレクトリは上書きしない」、15章、17.3の `game_id`

  - `--games N` はその番号から連続N個を作成する。
  - `game_state.json` の `game_id` は `game_XXXX` の番号（整数値）と一致させる。

## Q19. 乱数シード消費の厳密な順序と再現性ルール [解決済み → v0.5-draftに反映] **重大度: 中**
- **問題**: `--seed` を用いた完全再現性を実現するにあたり、乱数生成器の消費順序が一意に定まっていないと、実装（ループ順序や処理順）によって異なる結果が生成されてしまいます。
- **確認したいこと**: 1試合の中で、乱数生成器を消費する処理の実行順序を以下のように定義してよいか。
  1. **役職の割り当て**: 役職リスト `["werewolf", "seer", "villager", "villager"]` を乱数でシャッフルし、`config/agents.json` に定義されたプレイヤーの順（Claude -> Codex -> Grok -> agy）に順次割り当てる。
  2. **夜フェーズ（dry-run占い）**: 占い師に選ばれたプレイヤーが、自身を除く他3名から、乱数を用いて占い先を1名選出する。
  3. **昼フェーズ（dry-run投票）**: プレイヤーの順（Claude -> Codex -> Grok -> agy）に、各プレイヤーが自身を除く他3名から、乱数を用いて投票先を1名選出する。
  4. **同票処刑対象の選定**: 最多得票者が同票で複数名存在する場合、同票対象から乱数を用いて処刑対象を1名選出する。
- **決定**:
  - 上記1〜4の順で乱数を消費する。
  - 占い先、投票先、同票処刑対象の候補リストは、すべて `config/agents.json` の定義順を使う。名前のソート順は使わない。
  - フォールバックが必要な場合も、該当処理の時点で同じ候補順を使う。

## Q20. `game_state.json` の `executed` および `winner` の具体的な値 [解決済み → v0.5-draftに反映] **重大度: 低**
- **該当**: 17.3 game_state.json
- **問題**: 試合終了時の `executed` および `winner` のデータ型と具体的な値が定義されていません。
- **確認したいこと**:
  - `executed`: 処刑されたプレイヤーの名前（文字列。例: `"Grok"`, 同票時はランダム選出された1名）を格納する。
  - `winner`: 勝利陣営（文字列。`"werewolf"` または `"villager"`）を格納する。
- **決定**:
  - `executed` は処刑されたプレイヤー名の文字列とし、初期設定では `"Claude"`、`"Codex"`、`"Grok"`、`"agy"` のいずれかとする。
  - `winner` は `"villager"` または `"werewolf"` のみとする。
  - 人狼が処刑された場合は `"villager"`、人狼以外が処刑された場合は `"werewolf"` とする。

## Q21. 複数試合実行時における各試合の独立性 [解決済み → v0.5-draftに反映] **重大度: 低**
- **該当**: 14章、15.3章、20章 Phase 1
- **問題**: `--games N` 指定時、各試合の間で状態の引き継ぎがあるかどうかが明文化されていません。
- **確認したいこと**:
  - 各試合は完全に独立しており、前の試合の役職割り当てやログ等の状態は引き継がない。
  - 乱数生成器については、ゲーム全体の開始時に1つだけ初期化された乱数生成器インスタンスを全試合で引き継いで消費していくか、あるいは各試合 `i`（0〜N-1）の開始時に `seed + i` で個別に乱数生成器を初期化するかのいずれとするか（前者の「ゲーム全体で単一の乱数生成器インスタンスを使い回す」がシンプルで一般的）。
- **決定**:
  - 各試合は完全に独立させ、前試合の役職、占い結果、発言ログ、投票結果、処刑結果、勝敗、`game_state.json` の内容を引き継がない。
  - 乱数生成器は `run_game.py` 起動時に1つだけ初期化し、全試合を通して消費し続ける。
  - `--games 10 --seed 1234` では `random.Random(1234)` を1つ作り、全10試合で使い続ける。`seed + i` 方式は使わない。
