## Q1. 勝敗判定のタイミングと「2日目」の到達可能性 [解決済み → v0.3-draftでワンナイト人狼化（7章, 8章, 9章）]

**v0.3追記**: v0.2では「初日占い」を追加するワークアラウンドで対応したが、根本解決のため、v0.3-draftで初期版を**ワンナイト人狼**（夜の襲撃なし、生存者数による勝敗判定を廃止し、処刑結果のみで勝敗決定）に変更した。これにより1試合は夜フェーズ→昼フェーズ→処刑→勝敗判定の1周で必ず完結する設計となり、「2日目に到達しない」という問題はワークアラウンドではなく仕様そのものとして解消された。

- 該当: 7.2 (L132), 8. ゲーム進行 (L136-152)
- 4人村（人狼1・占い師1・村人2）で、7.2「人狼が生存したまま生存者数が2人以下」を文字通り適用すると、
  - Day1処刑で人狼が処刑されなければ生存者は3人 → 勝敗判定で継続
  - Night1で人狼が1名襲撃すると生存者は2人、かつ人狼生存 → 即人狼陣営勝利
  - 結果として、Day1で人狼を処刑できない限り、必ずNight1終了時点で人狼勝利が確定し、8章にある「2日目以降」のフェーズには実質到達しません。
- 占い師の占い結果（9. の「過去の占い結果」）も、そもそも2日目が発生しないなら次のDayフェーズで活用される場面がなく、仕様上の想定と矛盾している可能性があります。
- また、Night内の処理順（8.人狼の襲撃 → 9.占い師の占い → 10.勝敗判定）で、占いは襲撃結果確定後に行うのか、襲撃と同時（襲撃前の生存者情報を基準）に行うのかも未定義です。
- **確認したいこと**: 7.2の勝利条件は4人村用にこのままでよいか（＝実質1〜2ターンで終わる短期決戦を意図している）、それとも「生存者数が2人以下」は将来の6人・8人村向けの汎用ルールで、4人村では別の閾値（例: 人狼数 ≧ 村人陣営数）にすべきか。

## Q2. 昼フェーズにおける発言・投票の情報同時性 [解決済み → SPEC.md 10章]

- 該当: 9. 昼フェーズ (L161)
- 「生存している各プレイヤーが1回ずつ発言し、同時に投票先を決める」とありますが、11.1のJSON出力形式では `speech` と `vote` が同一レスポンス内で同時に生成されます。
- そのため、各AIは「自分より前に発言した他プレイヤーの当日の発言内容」を見た上で投票できるのか、それとも他プレイヤーの当日発言を一切見ずに自分の発言と投票を決めるのか（＝実質は前日までのログのみを根拠にした同時提出）が不明です。
- 呼び出し順（プレイヤーリスト順？ランダム？）も未定義です。
- **確認したいこと**: 発言は逐次公開（先に発言した人のログを後続プレイヤーに見せる）か、全員完全同時（当日の他人の発言は見えない）か。呼び出し順序の決め方は固定か毎タームシャッフルか。

## Q3. JSON構文が正しくても内容が不正な場合の扱い [解決済み → SPEC.md 12.2章]

- 該当: 12. JSONパース失敗時の処理 (L227-242), 13. 投票ルール
- 12章はJSONとして「構文的に」パースできない場合のフォールバックのみを定義しています。
- 構文的には正しいが内容が不正なケース（例: 存在しないプレイヤー名への投票・襲撃・占い、死亡者への投票、人狼が自分自身を`target`に指定、必須キー欠落、型不一致）の扱いが未定義です。
- **確認したいこと**: これらの意味的不正も12章のフォールバック（ランダム選択等）に含めてよいか。含める場合、フェーズごとの許容値チェック（生存者かどうか、自分以外かどうか等）をどこで実施するか（ゲームエンジン側でバリデーションしてから使う、という理解でよいか）。

## Q12. `seer_result` / 占い結果の構造と意味 [解決済み → v0.4-draftに反映]

- 該当: 9章、17.2、17.3（`seer_result: null` の例のみ）
- 問題:
  1. `game_state.json` の `seer_result` が `null` 以外でどんな形になるか未定義。占い対象・判定結果・占い師を保持できないと、昼プロンプト生成（10.3）や `results.md` の占い結果出力を一意に実装できない。

- **決定した方針**:

  - 占い結果は正確な役職名ではなく、人狼 / 人間の二値（`werewolf` または `human`）のみとする。`human` は非人狼を意味し、占い師と村人の区別はしない。

## Q13. 投票ラウンドの「同時決定」と公開タイミング [解決済み → v0.4-draftに反映]

## Q14. JSON検証の厳密さ（追加キー・空文字・前後テキスト） [解決済み → v0.4-draftに反映]

- 該当: 11章、12.1、12.2
- 問題: 必須キー欠落・型不一致・不正プレイヤー名は定義されているが、次が未定義:
  - 定義外キーの有無（`additionalProperties`）
  - 空文字（`""`）の `speech` / `vote` / `target` / `reason`

- CLIごとに応答形式がぶれやすく、実装者ごとに検証結果が変わりうる。
- **確認したいこと**: 応答全体を単一JSONオブジェクトとして厳密解析し、定義外キー・空文字・前後テキストは不正（フォールバック）とするか。それともコードフェンス除去などの緩いパースを許すか。
- **決定した方針**:
  - Phase 1では厳密JSONパースとする。応答全体が単一JSONオブジェクトでなければ不正扱いとする。
  - 定義外キー、必須キー欠落、空文字、null、型不一致、前後テキスト、Markdownコードフェンスはすべて不正（フォールバック対象）扱いとする。（緩いパースはPhase 3以降の拡張候補とする）

## Q16. `--seed` の適用範囲と再現性 [解決済み → v0.4-draftに反映]

- 該当: 12.3、13章、15.4
- 問題: `--seed` の存在は定義されているが、次が未定義:
  - 役職割当
  - 同票時の処刑対象
  - 投票失敗・占い失敗時のランダムフォールバック
  - `--games N` 時の試合間での乱数消費順
- **確認したいこと**: 同一コマンドライン・同一設定で完全に同じ試合結果を再現することをPhase 1の要件とするか。

## Q19. 乱数シード消費の厳密な順序と再現性ルール [解決済み → v0.5-draftに反映] **重大度: 中**

- **決定**:

- **問題**: CLASS.mdの当初のスタブは`build_speech_prompt(player, public_log)` / `build_vote_prompt(player, public_log)`だったが、`villager_prompt.md`/`seer_prompt.md`/`werewolf_prompt.md`のどれを連結するかは呼び出し時点のそのプレイヤーの役職に依存し、`{{seer_result_summary}}`も占い師かどうかに依存する。`PromptBuilder`自身はゲーム状態を保持しない設計のため、これらを引数で受け取る必要がある。
- **決定した方針**: `build_speech_prompt(player, role, public_log, seer_result_summary="")` / `build_vote_prompt(player, role, candidates, public_log, seer_result_summary="")` にシグネチャを拡張した。`build_night_prompt(seer, candidates)` は夜フェーズの行動者が常に占い師のため変更なし。CLASS.md 3.4章に反映済み。

### Q50. `common_player_prompt.md`内のMarkdownコードフェンス例示バグ [解決済み → prompts/common_player_prompt.md修正] **重大度: 中**

- **問題**: `common_player_prompt.md`の出力ルール説明文が「Markdownのコードフェンス（```json のようなもの）で囲まないでください。」となっており、禁止しているはずのコードフェンス記法自体を本文中に literal に含んでいた。これにより (1) `PromptBuilder`のテスト（`assert "```" not in prompt`）が失敗し、(2) AIへの指示として自己矛盾する内容になっていた。
- **決定した方針**: 該当行を「Markdownのコードフェンスで出力を囲まないでください。」に修正し、具体例の埋め込みをやめた。`tests/test_prompt_builder.py::test_real_prompts_do_not_contain_markdown_code_fences_in_body`で再発防止を検証する。

## Phase 3: AgentInvoker実装（2026-07-09）で判断した設計方針

`PromptBuilder` と `AgentInvoker` を組み合わせて `--use-real-agents` モードをサポートした際に発生した問題と、実装上の決定方針を記録する。

### Q51. Windows環境での subprocess.run デコードエラー (UnicodeDecodeError) 回避策 [解決済み → 実装に反映] **重大度: 高**
- **問題**: Windows環境では `text=True` を指定すると既定で CP932 (Shift-JIS) で標準出力をデコードしようとするが、AI応答やCLIエラーに UTF-8 文字が含まれていると `UnicodeDecodeError` が発生してゲーム全体がクラッシュする。
- **決定した方針**: `subprocess.run` 時に `encoding="utf-8"` および `errors="replace"` を明示的に指定し、文字化けや不正バイトを `\ufffd` に置換して文字列として安全にデコードする。

### Q52. 生応答 (raw) が NoneType になることによる TypeError 回避策 [解決済み → 実装に反映] **重大度: 中**
- **問題**: 例外発生時やデコード失敗時に `raw` 応答が `None` になる場合があり、そのまま `save_raw_response` を呼ぶと `TypeError: data must be str, not NoneType` が発生する。
- **決定した方針**: `save_raw_response` の呼び出し時および `AgentInvoker` の戻り値で `raw or ""` と空文字列にフォールバックさせ、`TypeError` を防止する。

### Q53. 一時ディレクトリの削除失敗時の挙動 [解決済み → 実装に反映] **重大度: 中**
- **問題**: 一時ディレクトリの削除（cleanup）に失敗した場合にゲーム進行を壊さないようにする具体的な実装。
- **決定した方針**: `AgentInvoker` 内で一時ディレクトリの削除エラーが発生した場合は例外を無視（警告の `warnings.warn` および `warnings` リストへの追加）し、ゲームエンジンが試合終了時に `pop_warnings()` で収集して `results.md` の `## 警告記録` 欄に安全に書き出す。

### Q54. CLI未導入環境での挙動 [解決済み → 実装に反映] **重大度: 高**
- **問題**: 環境に `claude` などのコマンドが全く存在しない場合の挙動。
- **決定した方針**: `FileNotFoundError` を `AgentCliError` （returncode 127）としてキャッチし、安全に `cli` エラーとして `raw/{seq}_{phase}_{player}_cli.txt` 保存＋フォールバック（ダミー応答）で進行させ、ゲームを完走させる。

### Q55. ルート直下ショートカット再導入案 [解決済み → 採用せず、Q33を維持] **重大度: 中**
- **問題**: v0.6-draft（Q33）では「Phase 1ではルート直下に`game_state.json`/`public_log.md`/`results.md`のコピーを作成しない」と確定していたが、Phase 3作業差分で実体コピー処理と`.gitignore`除外が追加され、Q33と実装が矛盾していた。
- **決定した方針**:
  - 再導入案は採用せず、Q33の「ルート直下3ファイルを作成しない」方針をPhase 3でも維持する。
  - `LogWriter` / `GameEngine`からルートコピー処理を削除し、保存先を`logs/games/game_XXXX/`配下だけに戻す。
  - ルート直下3ファイルを`.gitignore`へ追加せず、誤生成が`git status`で検出できるようにする。
- **注記**: Q30・Q31はQ33により引き続き前提消滅として扱う。
