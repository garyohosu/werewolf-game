# AIがAIを疑い、AIがAIに投票する：4つのAI CLIエージェントに「ワンナイト人狼」を遊ばせる実験基盤を作った話

## 1. はじめに

「AIが自分で考え、自律的に動き、他のAIと協力したり敵対したりする」――そんなSFのような世界が、今や個人のPC上で再現できる時代になりました。

最近ではさまざまなAIツールやCLIエージェントが登場していますが、「これらのAIを同じテーブルに並べて、非対称情報のゲームを遊ばせたらどうなるだろう？」という素朴な疑問から、本プロジェクトはスタートしました。

テーマに選んだのは、騙し合いの心理戦ゲームの代名詞である**「ワンナイト人狼」**です。
本記事では、**Claude / Codex / Grok / agy** という4つのAI CLIエージェントにワンナイト人狼を遊ばせ、勝率まで自動で集計できる実験基盤を構築した開発記録をお届けします。

---

## 2. 何を作ったのか

構築したのは、Pythonで動作する**「AI版ワンナイト人狼シミュレーター」**です。

### プレイヤー構成
- **ゲームマスター (GM)**: ゲームを進行させ、ログを記録・管理するスクリプト（この実験環境のオーガナイザーは **Claude Code** が務めています）
- **AIプレイヤー (4名)**: 
  - **Claude**
  - **Codex**
  - **Grok**
  - **agy**

### 基本ルール
ゲームは1晩・1日のみで決着する「ワンナイト人狼」ルールを採用しています。
- **役職構成**: 人狼 × 1、占い師 × 1、村人 × 2
- **進行フロー**: 
  1. 役職のランダム割り当て（夜フェーズ）
  2. 占い師による占い（夜フェーズ）
  3. 全プレイヤーによる発言ラウンド（昼フェーズ・順番発言）
  4. 全プレイヤーによる一斉投票ラウンド（昼フェーズ）
  5. 処刑者の決定と、陣営の勝敗判定

このシミュレーターを `--games 10` のようにコマンド一つで連続実行し、実行結果のJSONデータを解析して「どのAIがどの役職で何回勝ち、何回処刑されたか」を可視化できる集計プログラムを作りました。

## 3. なぜAIに人狼をさせるのか

多くのベンチマークテストでは、AIのロジックやコーディング能力、一般的な知識が評価されます。しかし、人狼ゲームをプレイするには、以下のような**「高次元の対人（対AI）能力」**が必要になります。

1. **非対称情報の処理**: 自分しか知らない秘密情報（占い結果や自分の役職）を抱え、他者とどう共有するか。
2. **意図の汲み取りと嘘の検出**: 先行プレイヤーの発言に含まれる矛盾や違和感を感知する。
3. **ゲーム理論的思考**: 人狼であれば「誰に擦り付けるのが最適か」、村人であれば「どうやって人狼を絞り込むか」の戦略。

この心理戦をAI同士で行わせることで、「AIがAIを本気で疑い、AIがAIを説得し、AIがAIに処刑投票を投じる」という非常に興味深いマルチエージェントの相互作用を観察できるようになります。

## 4. システム構成と安全設計

AI同士を戦わせるにあたり、最も重視したのは**「安全性（セキュリティ）」**と**「情報の非対称性の保持」**です。

### 情報漏洩の防止
人狼ゲームは「他のプレイヤーの役職を知らないこと」が大前提です。
そのため、シミュレーター内部の完全なゲーム状態（`game_state.json` など）は、各AIプレイヤーには一切見せない設計にしています。AIに渡されるのは、**「自分自身の役職」**と**「それまでに全員が発言した公開ログ」**、そして占い師であれば**「自分の占った結果」**のテキストのみです。

### サンドボックスによる実行環境隔離
各AIプレイヤーは外部CLIプロセスとして動的に呼び出されますが、以下の安全仕様を徹底しています。
- **一時ディレクトリでの隔離実行**: 外部CLIは、`tempfile.TemporaryDirectory` を用いてプレイヤーごとに都度生成される空の一時ディレクトリ（cwd）内で実行されます。これにより、外部AIにリポジトリ内のコードや他の試合結果ファイルを見せるリスクを排除します。
- **シェル不使用 (`shell=False`)**: 任意コマンドのインジェクションを防止するため、プロセス起動はシェルを介さずに行います。
- **タイムアウト設定**: AI応答が返ってこない、あるいは接続エラーでハングした場合に備え、60秒のタイムアウトを設けています。
- **不正応答のフォールバック**: AIが不正なJSONを返したり、コマンドが見つからなかったり、タイムアウトした場合でも、ゲームエンジンが「発言失敗」や「ランダム投票」などの代替行動（Fallback）に安全に流し、ゲーム全体をクラッシュさせずに必ず完走させます。

## 5. Phase 1〜4 の開発記録

本プロジェクトは、堅実なTDD（テスト駆動開発）とクリーンアーキテクチャの精神に基づき、4つのフェーズに分けて段階的に開発を行いました。

### Phase 1: Dry-Run エンジンの構築
まずは外部AIを実際に呼ばず、内部でシード付きの決定論的ダミー応答を返す `DryRunAgent` を実装しました。
これにより、役職割り当てから処刑、勝敗判定、およびエラー時のフォールバック処理が、乱数シードに基づいて100%再現可能な状態で正しく動作するゲームエンジンの基盤を固めました。

### Phase 2 & 2.5: プロンプト整備と `PromptBuilder`
各AIプレイヤーに送るプロンプトを整理しました。
「安全ルールの遵守（ファイル操作やシェル実行の禁止）」を指示する共通プロンプト、役職プロンプト、そしてフェーズ（占い・発言・投票）プロンプトを、プレイヤーの状態に合わせて動的に連結・プレースホルダ置換する `PromptBuilder` を実装しました。

### Phase 3: `AgentInvoker` による実CLI接続
`subprocess.run` を介して、各プレイヤーに対応する外部CLI（`claude`, `codex`, `grok`, `agy` など）を一時ディレクトリ上で安全に呼び出す `AgentInvoker` を実装。
Windows＋日本語環境でのデコードエラー（`UnicodeDecodeError`）を回避するためのエンコーディング設定や、一時ディレクトリ削除失敗時の例外処理など、プロダクションレベルの堅牢性を確保しました。

### Phase 4: `analyze_results.py` による勝率集計
各試合の完了後に `logs/games/game_XXXX/game_state.json` に保存される構造化データを一括パースし、全体の勝率を集計するプログラムを実装しました。

## 6. 実際に集計できる内容

作成した集計スクリプトを実行することで、以下のような統計レポートを瞬時に出力できます。

```bash
# デフォルトで全試合の統計をMarkdown形式で標準出力
python scripts/analyze_results.py

# JSON形式で結果を取得する（他の分析ツールとの連携用）
python scripts/analyze_results.py --format json

# 指定されたファイルに集計レポートを保存
python scripts/analyze_results.py --output reports/summary.md
```

### レポート出力例 (Markdown)

```markdown

- Valid games: 21

- Werewolf wins: 14
```

この基盤があることで、「今回は人狼陣営の勝率が著しく高い」「Grokは人狼の時に勝率が良いが、Claudeは占い師の時に勝率が良い」といった傾向（※あくまで本実験環境・モデル設定における一例です）を、客観的な数値データとして評価・観察することができます。

## 7. 開発中の気づきとAIツール体験

今回の開発は、私自身が手でコードを書くだけでなく、開発パートナーとしての **Claude Code** や **Codex CLI** と対話しながら進めました。

特に印象的だったのは、AIエージェントの進化です。
開発中に「Windowsの標準出力デコードエラーでプロセスが落ちる」という複雑な環境依存の不具合が発生した際、AIエージェントは自動的にエラーログをパースし、`universal_newlines` 実行時に `encoding="utf-8", errors="replace"` を追加するという回避コードを瞬時に提案してくれました。

単に指示に従ってプログラムを作るだけでなく、自律的にテストを書き、失敗したテストの原因を突き止め、修正してプッシュするまでのサイクルを、驚くべき精度で並行して回すことができました。

## 8. 今後やりたいこと（今後の展望）

本シミュレーターはまだ「実験基盤」の第一歩です。今後は以下のような拡張を考えています。

1. **Phase 5: 記事化用の詳細ログ抽出**: 試合の履歴から「AIによる面白い煽り合い」や「劇的な騙しコード」を別のLLMで自動ピックアップし、note記事の素材を自動生成する。
2. **通常ルールへの拡張**: 襲撃あり、複数夜・複数日進行の8人村・10人村への拡張（人狼の夜襲撃、狂人や狩人などの役職追加）。
3. **ペルソナ（性格プロンプト）の追加**: 「感情的に反論するAI」「論理的に詰め寄るAI」など、AIにプレイスタイルを与えて挙動の変化を観戦する。
